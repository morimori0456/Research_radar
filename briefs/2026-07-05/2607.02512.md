# Program-as-Weights: A Programming Paradigm for Fuzzy Functions

- **arxiv-id**: 2607.02512 ／ **topic**: sns_wildcard ／ **project**: EXPLORE (P2 に強く接続) ／ **relevance**: 4/5 ／ **hf_upvotes**: 68
- **URL**: https://arxiv.org/abs/2607.02512

## 何を解決する論文か
「重要なログ行への警報」「壊れた JSON の修復」「意図に沿った検索結果の順位付け」のような、明確なルールで書けない“ふわっとした”処理は、いま LLM API に丸投げされがちで、その代償として locality(手元で完結する性質)・再現性・コストを失っている。本論文は、そうした処理を自然言語の仕様から **compact でローカル実行可能なニューラル成果物へコンパイルする** fuzzy-function programming を提案する。foundation model を「入力ごとに解く問題解決器」ではなく「関数定義ごとに一度だけ呼ぶ道具作り(tool builder)」として捉え直すのが核心。

## 手法の核心
- **Program-as-Weights (PAW)**: 4B の compiler モデルが自然言語仕様を受け取り、凍結された軽量 interpreter 用の **parameter-efficient adapter (少量の追加パラメータだけで挙動を差し替える小さな重み。LoRA 等と同系)** を直接“emit”する。関数ごとにこの adapter が1つの再利用可能な成果物になる。
- **FuzzyBench (10M 例) を新規公開**: この compiler を訓練するためのデータセット。仕様→挙動の対応を大量に学習させる土台。
- **凍結 interpreter + 差し替え adapter という二層構成**: 0.6B の Qwen3 interpreter を固定したまま、adapter だけで関数の中身を切り替える。推論本体は共有し、関数差分だけを軽量重みに閉じ込める。
- **capacity gap を跨ぐ結果**: PAW を実行する 0.6B interpreter が、32B (Qwen3-32B) への direct prompting と同等性能に到達。推論メモリは約 1/50、MacBook M3 上で 30 tokens/s。
- **コスト構造の転換**: compiler は関数定義ごとに1回だけ走り(高コスト)、以後の各呼び出しは安価かつオフライン。「1回コンパイル→何度も安く offline 実行」という償却モデル。

## P2(Foundation Model 蒸留 + 適合)への適用可能性
- **“大教師の能力を小生徒の軽量重みに封じ込める”という点で、まさに蒸留/PEFT の変種**。ただし通常の蒸留(教師出力を生徒に模倣させる)と違い、**教師(compiler)が生徒の重み(adapter)を直接生成する** amortized な方式で、R2「capacity gap 系統則」に新しい軸を与える。
- 0.6B が 32B に並ぶという **capacity gap を跨ぐ具体例**として、自社の大教師→小生徒蒸留における「どこまで縮められるか」の上限探索の参照点になる。
- 「foundation model = tool builder」という枠組みは、エッジ配備(本ループの Jetson/MacBook 級環境)で **推論を offline・低メモリに落とす適合レシピ**として直接効く。
- 注意: 対象タスクは log 警報・JSON 修復・検索順位付けなどの汎用テキスト処理で、AD ドメインとの距離はある。あくまで“蒸留/適合の方法論”としての学びに価値がある探索枠。

### 試すべき実験アイデア
1. **hypernetwork 型蒸留の予備検証**: PAW の「compiler が adapter 重みを直接吐く」設計を、自社の小規模タスク(例: 特定シナリオ判定の軽量分類ヘッド)で再現し、通常の feature/output 蒸留(cf. 2607.01851)と *同一 student サイズ・同一データ*で精度/メモリを比較。R2 の「教師/生徒比 vs 破綻境界」の点を1つ増やす。
2. **adapter emit vs LoRA fine-tune のコスト対効果**: 同じ仕様を (a) PAW 風の1回 compile で adapter 生成、(b) 少データ LoRA 微調整、で作り、生成コスト・推論メモリ・精度を三軸プロット。「適合を forward 1回に amortize できるか」を測る。
3. **capacity gap の走査**: interpreter を 0.6B / 1.5B / 4B と振り、compiler 由来 adapter で埋められる gap の限界を測定し、direct prompting の大モデルと交差する点を探す(R2 の系統則づくりの素材)。

## メモ
FuzzyBench(10M 例)が公開されれば、蒸留/適合レシピの再現実験に使える外部データとして価値がある。「amortized adapter generation(仕様→重みを1回の forward で生成)」という筋は、P2 の翌日以降の探索テーマとしても記録に値する。
