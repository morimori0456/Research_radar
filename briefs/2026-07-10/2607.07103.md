# K-Risk: LLM アノテーション付き高リスク運転シナリオの知識拡張データセット

- **arxiv-id**: 2607.07103 ／ **topic**: planner_ai ／ **project**: P1 ／ **relevance**: 4.5/5 ／ **hf_upvotes**: 0
- **URL**: http://arxiv.org/abs/2607.07103v1

## 何を解決する論文か
安全な自動運転には、頻出する高リスク事象への即応と、稀な long-tail シナリオ (自然な走行データにはほとんど現れない極端な事象) への深い推論の両方が必要だが、既存の軌道データセットには高リスク事象のラベル・意味的アノテーション・検証可能な安全シグナルがほぼ付いていない。本論文は、構造化された走行軌道と LLM 生成の意味的アノテーションを組み合わせた高リスクシナリオ特化データセット **K-Risk** を公開し、risk-aware な自動運転エージェントの開発・評価の標準基盤を提供する。

## 手法の核心
- **20 データセットの統合**: 欧州・中国・米国の人間運転 + 自動運転車の軌道データセット 20 個を統合し、highway・都市高速・交差点・roundabout をカバー。単一ソースの偏りを避けたリスクイベント抽出の母集団を作る。
- **risk-centric 抽出パイプライン**: 統一されたリスク中心の抽出処理で **31,398 件の高リスク事象**をキュレーションし、さらに near-collision (衝突寸前) ケース **1,036 件の extreme サブセット**を切り出す。
- **同期された triplet 形式**: 各事象を「軌道 + メタデータ + 言語」の同期 triplet として提供。言語部分は構造化シナリオ記述・異常挙動の通知・(代表サブセットには) 因果的リスク分析と行動推奨を含む。
- **closed-loop simulator による検証**: LLM 生成の因果分析と行動推奨は、closed-loop simulator (エージェントの行動が環境に反映され、その結果が次の入力になる閉ループ型シミュレーション) 上で iterative reflection (生成→検証→修正の反復) を回して妥当性を検証済み。「LLM が書いただけ」のアノテーションと一線を画すのはこの検証工程。

## P1 (Planner AI + 評価) への適用可能性
- relevance_criteria の「ベンチマーク/評価手法の新提案は特に高優先」に直球。planner の安全性評価は「リスクが顕在化するシナリオをどれだけ持っているか」で決まるため、31k 件のラベル付き高リスク事象 + 1k 件の near-collision サブセットは評価シナリオバンクとしてそのまま価値がある。
- 軌道と言語が同期している点は、評価レポートの自動生成 (「なぜこのシナリオで planner が減点されたか」の言語化) や、LLM ベースの評価器の学習データとしても使える。
- 懸念は LLM アノテーションの品質が closed-loop 検証済みなのは「代表サブセット」のみである点と、日本の道路環境が 20 データセットに含まれない可能性。relevance 4.5 (5 でない) の理由。

### 試すべき実験アイデア
1. **評価シナリオバンクへの取り込み**: extreme サブセット 1,036 件を自社シミュレーション形式に変換し、既存 planner の closed-loop 評価に流して失敗率・失敗モードの分布を取る。既存シナリオ集で見えていなかった failure mode が出るかが最初の検証点。
2. **リスク抽出パイプラインの流用**: K-Risk の risk-centric 抽出基準 (TTC 等の指標がどう定義されているか論文本体で確認) を自社走行ログに適用し、自前の高リスク事象マイニングを同じ基準で回せるか試す。
3. **言語アノテーションの評価器学習**: 「シナリオ記述 → 因果リスク分析」ペアで小型 LLM を fine-tuning し、自社シナリオの自動リスク説明生成が成立するか検証する (P2 の適合 recipe とも接続)。
