# LongStraw: Long-Context RL Beyond 2M Tokens under a Fixed GPU Budget

- arXiv: https://arxiv.org/abs/2607.14952
- id: 2607.14952 / published: 2026-07-15 / hf_upvotes: 159
- 選別枠: sns_wildcard (EXPLORE) — 本日の探索枠 1 件

## 何を解決する論文か

推論側の context 長 (100万トークン級) と、RL post-training (強化学習による事後学習) 側の context 長 (256K 以下が普通) の間に開いた差を埋める研究。エージェントは観測・tool 出力・過去の判断が軌跡上に積み上がるため長い context を要求するが、RL 学習は活性値を全部保持する必要があり、そこがメモリの壁になる。LongStraw は GPU 台数を増やさずに 200 万トークン超の RL post-training を「実行可能にする」実行スタックを提案する。

## 手法の核心

- **GRPO (Group Relative Policy Optimization; PPO を簡略化した、LLM の RL 学習で使われる手法。1つのプロンプトに対し複数の応答を生成し、その group 内の相対的な優劣を advantage として使うので value network が要らない) を前提に据える。** group 内で prompt が共有されるという構造をメモリ削減に使い切るのが全体の発想。
- **共有 prompt 部分を autograd なし (勾配計算グラフを作らずに) 評価する。** 長大な context の大半は group 内で共通なので、そこに学習グラフを持たせない。
- **後続トークンが必要とする model-specific state だけを残す。** 全活性値ではなく、再計算に不可欠な状態のみ保持する。
- **応答ブランチを1本ずつ replay する。** 同時に持つ学習グラフを小さくする代わりに、replay の追加計算時間を払う (メモリと時間のトレードを明示的に選んでいる)。
- **アーキテクチャを意識した実装。** hybrid recurrent + full-attention の Qwen3.6-27B と、compressed-attention MoE (Mixture-of-Experts; 入力ごとに一部の専門家サブネットだけ活性化する構造) の GLM-5.2 の2種で実装。

## 報告されている数値

- 8×H20: group size 2 と 8 で 2.1M positions の scoring + backward を完走。**group size を 2→8 に増やしても peak allocated memory の増分は 0.21 GB のみ。** 別のストレステストで 4.46M positions 到達。
- 32×H20: 2.1M トークンの prompt を GLM-5.2 の全 78 層でエンドツーエンド実行。

## 重要な留保 (読むときに外せない点)

著者自身が abstract で、これらの実験は **"execution capacity rather than complete training correctness" (実行できることを示したのであって、学習が正しく回ることを示したのではない)** と明記している。捕捉した prompt state は detach されており、分散環境での forward / 勾配合成の一部経路は未完成。**つまり「2M トークンで RL が回って性能が出た」という論文ではなく、「メモリ的に実行可能になった」というシステム論文。** 精度・収束の主張として引用しないこと。

## 追っているプロジェクトへの適用可能性

**P2 (FM 蒸留 + 適合) — 主に効く。**
自社で長い軌跡を持つモデルを post-training する際、「context 長を諦める」か「GPU を増やす」かの二択だったところに第三の選択肢を与える。特に group size をほぼ無料で増やせる性質は、GRPO 系の分散低減にそのまま効く。ただし上の留保があるので、まずは**再現ではなく設計の借用**として読むのが正しい距離感。

**P3 (次世代アーキテクチャ) — 間接的。**
「共有 prefix を autograd 外に置く」という発想は、運転 E2E で長い走行文脈を扱うときの学習側コスト設計に転用できる。RoboTTT (2026-07-18 ブリーフ) が推論側で長 context を fast weights に圧縮したのと**対になる**議論 — あちらは推論、こちらは学習。並べて読むと「長 context をどこでコストを払って畳むか」の設計空間が見える。

**P1 — 直接の関連は薄い。**

## 試すべき実験アイデア

1. **小規模での安全な追試:** 手元の 7B 級モデルで、GRPO の group size を 2→8→16 に上げたときの peak memory 増分を計測し、論文の「group size はほぼ無料」が自分のスタックでも成り立つか確認する。ここは correctness の留保に触れない範囲で検証できる。
2. **replay コストの実測:** メモリ削減と引き換えの replay 時間が、実際の wall-clock でどれだけ効くか。「メモリは減ったが遅すぎて使えない」が現実的な失敗モードなので、採否はここで決まる。
3. **prefix 共有率との関係:** 自社の agentic 軌跡で、prompt のうち group 内共有部分が何割かを先に測る。共有率が低いタスクだと本手法の利得はほぼ消えるはずで、**これは導入前に 1 日で測れる足切り指標**。
4. **RoboTTT との対比整理:** 長 context を「学習時に replay で畳む」vs「推論時に fast weights で畳む」の比較表を作り、P3 のアーキ選定メモに落とす。
