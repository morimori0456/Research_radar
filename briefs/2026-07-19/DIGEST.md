### LongStraw: Long-Context RL Beyond 2M Tokens under a Fixed GPU Budget (EXPLORE, up=159)

推論側が 100 万トークンの context を扱えるようになった一方、RL post-training (強化学習による事後学習) は 256K 止まりという**学習と推論の非対称**を、GPU を増やさずに埋めにいく systems 論文。カギは GRPO (Group Relative Policy Optimization; 1つのプロンプトに複数の応答を生成し group 内の相対優劣で学習する、PPO の簡略版) で **prompt が group 内で共有される**という構造を突き、共有部分を勾配計算グラフの外に出して、応答ブランチだけを1本ずつ replay すること。結果として group size を 2→8 に増やしてもピークメモリ増分は 0.21 GB に留まる。

## パイプライン障害の報告 (本日の主要な発見)

今日の実質的な成果物は論文 1 本より、**収集側が壊れている証拠が揃ったこと**にある。

### 観測された事実

1. `candidates.json` の `generated_at` は 2026-07-18T18:00Z (= 07-19 03:00 JST)。**ファイル自体は今朝の実行で正しく生成されている** (stale file ではない)。
2. にもかかわらず **arXiv 由来の候補が全トピックで 0 件**。3 件すべてが Hugging Face Daily Papers 由来の wildcard。
3. cutoff は `lookback_days=2` より 2026-07-16 18:00Z (木)。**金曜 07-17 の arXiv announcement は本来この窓に入る**ため、「週末で新着がない」では説明がつかない。
4. 拾われた 3 件の published は 07-13 / 07-15 / 07-13 で、**いずれも cutoff より古い**。HF 経路が cutoff 判定を通っていないことの裏付け。
5. `briefs/2026-07-17/` が空ディレクトリ。同じ障害が既に一度起きている可能性。

### 推定原因 (fetch_candidates.py)

- **arXiv API 呼び出しの失敗が握り潰されている** — `fetch_candidates.py:138` の `except Exception` が `entries = []` にフォールバックし、`[fetch] 0 new` と出力する。**「API が落ちた」と「新着が無い」が出力上まったく区別できない。** 3 トピック全滅かつ金曜分が窓内という状況から、これが本命。
- **wildcard 経路に重複除去も日付フィルタも無い** — `fetch_candidates.py:160-172` の wildcard 追加は、その日の arXiv 候補 (`base_ids`) との重複しか見ておらず、**過去に処理済みの論文を除外する仕組みが無い**。HF の upvote 上位は数日間ほぼ変わらないため、arXiv 側が 0 件の日は**前日と同じ wildcard がそのまま再提示される**。今日 2/3 が重複したのはこれで完全に説明できる。これは以前から疑っていた重複問題の実体でもある。

### 推奨する修正 (未実施 — 指示待ち)

1. **fetch の失敗を可視化する。** API エラー時は 0 件で握り潰さず、`candidates.json` に `errors: [...]` を残すか、全トピック 0 件なら非ゼロ終了する。今のままでは障害が静かに続く。
2. **処理済み ID の永続化。** `briefs/*/` を走査するか `seen_ids.json` を持ち、wildcard 追加時に過去分を除外する。
3. **wildcard にも cutoff を適用する** か、少なくとも published を見て古すぎるものを落とす。
4. **リトライ。** arXiv API の一過性障害に対し指数バックオフで 2-3 回。

修正するか、まず数日ログを取って原因を確定させるかは判断を仰ぎたい。

## プロジェクト別の要点

### P1: Planner AI + 評価
**本日の入力なし。** 候補 0 件 (収集障害のため、落選判定ではない)。空白日として扱い、07-18 の DRIFT のフォローが未着手ならそちらを優先するのが妥当。

### P2: FM 蒸留 + 適合
LongStraw が間接的に関連。**GRPO の group size をほぼ無料で増やせる**という性質は、社内で GRPO 系の post-training を回す際の分散低減にそのまま効く可能性がある。ただし correctness 未検証なので、まず「自社の軌跡で prompt の group 内共有率が何割か」を測るのが低コストな足切り (共有率が低ければ利得はほぼ消える)。

### P3: 次世代アーキテクチャ
LongStraw は 07-18 の RoboTTT と**対になる**論点を提供する。RoboTTT は長い context を**推論時**に fast weights (推論時にも勾配で更新される重み) へ圧縮し、LongStraw は**学習時**に replay で畳む。「長 context のコストをどの段階で払うか」という設計空間の両端として並べて整理する価値がある。
