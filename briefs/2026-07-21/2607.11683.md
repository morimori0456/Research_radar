# RAGU: A Multi-Step GraphRAG Engine with a Compact Domain-Adapted LLM

## 何を解決する論文か

GraphRAG (Graph Retrieval-Augmented Generation; 文書から knowledge graph を作り、その構造化知識で LLM の回答を補強する RAG の一種) は、従来「1 パスで一気に entity/relation を抽出」するため、ノイズだらけの entity と壊れやすい retrieval を生む。RAGU はこの抽出工程を **抽出と統合 (consolidation) に分離**し、さらに「抽出役の LLM は巨大である必要はない」という仮説のもとに **7B の小型モデル Meno-Lite-0.1** を専用に訓練して、KG 構築で Qwen2.5-32B を上回った、という主張の論文。

## 手法の核心

- **抽出と統合の分離パイプライン:** two-stage typed extraction (型付きで entity/relation を 2 段抽出) → DBSCAN (密度ベースのクラスタリングで、しきい値ではなく近さで塊を作る手法) による重複除去 → LLM summarization → Leiden community detection (グラフを密に繋がった部分集合=community に分割するアルゴリズム)。1 パス抽出のノイズを後段で洗う設計。
- **中核の仮説 (ここが P2 に効く):** in-pipeline の LLM に必要な能力=comprehension・extraction・reasoning-over-context は **"language skills" であり、モデルサイズに対して弱くしかスケールしない**。一方 factual world knowledge (事実知識) はサイズに強く依存する。→ **抽出役に大モデルは要らない**という主張。
- **compact extractor の実証:** language skills に最適化した 7B の **Meno-Lite-0.1** が、KG 構築で Qwen2.5-32B に **+12.5% relative harmonic mean**、英語 GraphRAG タスクでは同等。単一 GPU で動作。
- **retrieval 品質:** GraphRAG-Bench (Medical) で evidence recall が全 factoid level で最良 (最大 0.84 vs 既存 ≤0.76)。synthesis タスクで HippoRAG2 超え。
- **反証的な指摘:** multi-hop factoid QA での「HippoRAG2 が強い」という既存の見え方は、実は **answer-format artifact** (回答の書式の違いによる見かけの差) が大きいと分析。ベンチ数値を額面で信じない姿勢。
- pip 導入可 (`pip install graph_ragu`)、MIT、コード・重み公開。

## 追っているプロジェクトへの適用可能性

**P2 (Foundation Model 蒸留 + 適合) — 本命。** この論文の価値は RAG エンジンそのものより、**「どの能力がサイズに依存し、どれが依存しないか」を能力分解した点**にある。蒸留は結局「student にどの能力まで移すか」の取捨選択であり、この分解は student のターゲット選定と capacity gap (teacher と student の表現力差が開きすぎて蒸留が失敗する現象) の見立てに直接効く。
- **試すべき実験アイデア 1:** 自分たちの student タスクを「language skill 寄り (抽出・整形・context 追従)」と「world-knowledge 寄り (事実想起)」に分類し、前者だけを狙った小型 student が、汎用蒸留した同サイズ student をどれだけ上回るか A/B。仮説が正しければ、**同じパラメータ予算をタスク特化に振る**方が得。
- **試すべき実験アイデア 2:** ドメイン適合の観点で、大 teacher からの汎用蒸留 vs. Meno-Lite 流の「language-skill 特化訓練 + ドメイン語彙だけ後付け」を比較。少データ域でどちらが速く適合するかを測る (P2 の「少データ適合」評価軸に一致)。
- **試すべき実験アイデア 3:** 「answer-format artifact」の指摘を自分たちの蒸留評価にも適用 — student の劣化が本当に能力差なのか、出力書式の不一致なのかを切り分ける ablation を評価パイプラインに組み込む。

**P1 (Planner 評価):** 直接の転用は薄い。ただし「見かけの優劣が format artifact だった」という**評価の落とし穴の指摘**は、closed-loop メトリクスの比較でも警戒すべき一般教訓。

**P3 (次世代アーキ):** 低め。MoE でも新規構造でもなく、寄与は「能力分解 × 小型特化」という設計思想レベル。

## 留保

- abstract 由来の数値のみで、7B が 32B を超えるのは **KG 構築という狭いタスク**での話。汎用能力での逆転を主張しているわけではない点に注意。
- 「language skills はサイズに弱依存」は魅力的だが、本文でどう能力を切り分けて測ったか (評価設計) を確認しないと、蒸留戦略の根拠には使えない。
