# DRIFT: Drift and Aggregation for Motion Planning

**arXiv:** 2607.14507 | **topic:** planner_ai (P1) | **score:** 4.5/5 | **hf_upvotes:** 0

## 何を解決する論文か

End-to-end のプランナーは「複数のもっともらしい運転挙動」を表現しつつ、実時間制約下で単一の実行可能な軌道を1本出す必要がある。proposal-based 手法 (複数の候補軌道を生成して選ぶ方式) は多峰性を扱えるが、候補集合をどう最終案に畳むかが未解決の設計問題として残る。DRIFT は「compact な軌道 latent 空間での one-step drifting + scene-aware な proposal 集約」で、候補品質の教師ラベルなしにこの畳み込みを学習する。

## 手法の核心

- **one-step drifting**: 拡散のような多ステップ denoising ではなく、軌道 latent 空間で1ステップの drift (連続的な微小移動を1回で行う生成操作) により候補を生成。DRIFT Decoder が pretrained visual encoder の特徴を条件に、48 個の proposal 特徴を**1回のバッチ推論**で出す (alpha=0.5 で32本、alpha=0.9 で16本。alpha は生成の多様性/保守性を振るノイズ強度)
- **Aggregation Head**: 軽量ヘッドが proposal 特徴と scene / navigation / ego-state を統合し、最終軌道を直接回帰。**proposal ごとの品質ラベルを必要としない** (集約の学習が教師なしで済むのが実務上の利点)
- **map-derived boundary regularizer**: drivable polygon (走行可能領域) の外の waypoint と、内側でも境界に近すぎる waypoint を罰する正則化で、地図整合性を明示的に付与
- **NAVSIM navtest** で PDMS 89.6 / EPDMS 90.4 (PDMS = Predictive Driver Model Score; NAVSIM の閉ループ相当スコア。EPDMS はその拡張版)。drivable-area 遵守と ego progress が強い
- 速度: proposal 生成+集約が RTX 4090 で **10.82 ms**、visual backbone 込みの全体で 66.43 ms

## P1 への適用可能性と実験アイデア

- 追っている planner 評価パイプラインに対し、DRIFT は「多仮説を安く出し、ラベルなしで1本に畳む」実装レシピとして直接輸入候補。特に **proposal 品質ラベルが不要**な点は、社内で軌道品質のアノテーションが乏しい状況に噛み合う
- 実験: 現行プランナーの後段を DRIFT の Aggregation Head 型 (proposal 特徴を直接回帰) に差し替え、NAVSIM の EPDMS と推論時間で A/B。特に boundary regularizer の有無で drivable-area 違反率がどれだけ動くかを切り分ける
- 評価軸として、alpha を振った proposal 集合の多様性が、交差点/合流など多峰なシーンで EPDMS のどの下位指標 (collision, progress, comfort) を改善するかを分解して見る
