# Harness Handbook: Making Evolving Agent Harnesses Readable, Navigable, and Editable

**arXiv:** 2607.13285 | **topic:** sns_wildcard (EXPLORE) | **score:** 4/5 | **hf_upvotes:** 186

(本日の探索枠。分野外だが hf_upvotes 186 と突出した注目度で、かつエージェント基盤を運用する実務者にとって学びの価値が高いため採用)

## 何を解決する論文か

現代の AI エージェントの能力は foundation model だけでなく **harness (プロンプト構築・状態管理・tool 呼び出し・実行調整を担う周辺コード基盤)** にも依存する。モデル・API・環境・要件が変わるたび harness は改修が要るが、その前に「目的の挙動を実装している全コード箇所」を特定せねばならない。本番 harness は巨大で密結合、挙動がコード全体に分散しているため、これが困難。本論文は **behavior localization (挙動→コード箇所の対応付け)** を harness 進化の中心的ボトルネックと位置づけ、その自動化表現を提案する。

## 手法の核心

- **Harness Handbook**: harness のコードベースから、静的解析 + LLM による構造化で**自動生成**される behavior-centric な表現。各「挙動」をそれを実装するソース箇所に紐づける
- **BGPD (Behavior-Guided Progressive Disclosure; 高レベル挙動から実装詳細へ段階的に開示する誘導方式)**: エージェントを高レベル挙動→関連実装へ誘導し、候補箇所を現行ソースに照合して検証する
- 2 つの OSS harness の多様な改修要求で、Handbook 支援の計画立案が behavior localization と edit-plan の質を改善し、しかも planner の消費 token は少ない
- 効果は挙動が散在 (scattered) するケースで最大 — まさに密結合コードで人手が詰まる場面

## 学び (分野外だが / 研究ループ運用への示唆)

- この research_loop 自体が「evolving harness」(fetch_candidates.py・各 review スクリプト・topics.yaml が絡む調整層) であり、本論文の主張は自分たちのパイプライン保守に直接応用できる。「トピック選別の挙動はどこか」「dedup ロジックはどこか」といった behavior→code の対応を Handbook 化しておくと改修が速くなる
- 実験アイデア: 現行の fetch/選別スクリプト群に対し、挙動 (候補取得・dedup・quota 制御・brief 生成) を behavior-centric に列挙し各ソース箇所へ手動リンクした軽量 Handbook を作る。次に改修要求 (例: [[fetch-duplicate-candidates]] の dedup 強化) を出したとき、Handbook 有無で localization の手数がどれだけ減るかを比較
- より一般に、社内でエージェント/自動化パイプラインを増やすほど「behavior localization コスト」が支配的になるという視点は、P2/P3 の実験基盤を組む際の設計原則として持っておく価値がある
