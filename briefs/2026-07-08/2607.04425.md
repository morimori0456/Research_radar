# UI-MOPD: Multi-Platform On-Policy Distillation for Continual GUI Agent Learning

- **arxiv-id**: 2607.04425 ／ **topic**: sns_wildcard ／ **project**: EXPLORE (P2 に接続) ／ **relevance**: 3/5 ／ **hf_upvotes**: 62
- **URL**: https://arxiv.org/abs/2607.04425

## 何を解決する論文か
GUI agent (PC やスマホの画面を見て操作を実行するエージェント) を複数プラットフォーム (OS・モバイル等) に対応させると、プラットフォームごとに操作作法が違うため、joint 学習や逐次学習では **挙動パターンの混合・プラットフォーム固有能力の劣化・catastrophic forgetting (新タスク学習で旧能力を忘れる現象)** が起きる。本論文は multi-teacher の on-policy distillation を continual learning に持ち込み、新プラットフォーム適応と既存能力保持を両立させる。

## 手法の核心
- **on-policy distillation** (teacher のデモ軌跡ではなく、**student 自身が到達した状態の上で** teacher の出力を教師信号にする蒸留。訓練時と実行時の状態分布ミスマッチを減らす) を GUI agent の continual learning に初適用。
- **multi-teacher + 動的選択**: プラットフォームごとに platform-specific teacher を用意し、現在の環境に応じて **どの teacher から蒸留するかを動的に切り替える**。
- **platform-conditioned distillation**: teacher の挙動事前知識を、プラットフォーム条件付けを施した上で単一の shared policy へ転移する。作法の混合を条件付けで分離するのが肝。
- **Uni-GUI dataset**: 実行可能なクロスプラットフォーム操作軌跡の高品質データセットを新規構築。
- **結果**: OSWorld 38.2% / MobileWorld 12.0% のタスク成功率で、クロスプラットフォーム能力の保持と新プラットフォーム適応のバランスを実証。

## P2 (Foundation Model 蒸留 + 適合) への適用可能性
- 探索枠だが問題構造は P2 の本題と同型: 「**ドメイン (プラットフォーム) ごとに専門化した teacher 群 → ドメイン条件付けした単一 student への on-policy 蒸留**」は、P2 の「他機種・他ドメインへの適合」— 車種・センサ構成ごとの専門モデルを単一の基盤モデルへ統合しつつ、機種追加時に既存機種の性能を落とさない — にそのまま写像できる。
- on-policy 化のポイント (student の到達する状態分布上で教師を照会する) は、蒸留した運転モデルが自分の癖で到達する状態での挙動保証、という自動運転でも既知の課題 (DAgger 系の発想) と一致し、recipe として素直に移植できる。

### 試すべき実験アイデア
1. **機種条件付き蒸留の試作**: 車種 A/B それぞれの fine-tuned teacher から、機種 ID を条件入力に持つ shared student へ蒸留し、機種 C 追加時の A/B 性能保持率を「teacher 切替なし joint 蒸留」と比較する。
2. **on-policy vs off-policy 蒸留の比較**: teacher デモ上での蒸留と、student rollout 状態上での蒸留 (closed-loop シミュレータで実施) を比較し、分布ミスマッチ由来の性能差を定量化する。
3. **忘却の測定基盤**: 機種追加のたびに全既存機種の評価スイートを自動で回し、catastrophic forgetting を継続監視するダッシュボードを P2 パイプラインに組み込む。
