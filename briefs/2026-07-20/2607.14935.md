# VideoChat3: Fully Open Video MLLM for Efficient and Generalist Video Understanding

## 何を解決する論文か

動画を理解する MLLM (Multimodal Large Language Model; 画像や動画などをテキストと一緒に扱える大規模言語モデル) のオープンソース実装が抱える3つの弱点 — 「特定ドメインでしか使えない」「計算コストが高い」「学習コードやデータが非公開で再現できない」— をまとめて潰しにいく論文。VideoChat3 は**効率**と**汎化**を両立させた上で、モデル重みだけでなく学習コード・学習戦略・データセットまで公開する "fully open" を標榜する。「動画版の再現可能なベースライン」を業界に置くことが実質的な狙い。

## 手法の核心

- **I3D-ViT (Inflated 3D Vision Transformer; 画像用に学習済みの 2D ViT の重みを時間軸方向に「膨らませて」3D に転用する手法。動画用エンコーダをゼロから学習せずに済む) を視覚エンコーダに採用。** 時空間表現を安く獲得するのが狙いで、動画エンコーダの学習コストを回避する古典的だが効く設計。
- **Adaptive Frame Resolution for Streaming Video Perception。** ストリーミング入力に対しフレームごとの解像度を適応的に変える。全フレームを一律の高解像度で処理しない = 学習・推論の両方で入力トークン数を削る。**「どのフレームに計算を割くか」を動的に決める発想**で、効率の主要因はここ。
- **スケーラブルな動画データ合成パイプライン。** 人手アノテーションに依存せず学習データを生成し、性質の異なる3つのデータセットを整備する。
- **用途別に3データセットを分けて公開:** `VideoChat3-Academic2M` (一般的な動画理解 200万件規模)、`VideoChat3-LV116K` (long-form; 長尺動画)、`VideoChat3-OL617K` (online/streaming; 逐次到来する動画)。**「一般 / 長尺 / ストリーミング」という切り分け自体が、汎化を設計で担保しにいく構造。**
- **fully open。** 重み・学習コード・学習戦略・データの全公開。partially open な既存モデルとの差別化点であり、後述のとおり実務上はここが一番大きい。

## 読むときの注意

abstract は効率と汎化の**両立**を主張するが、公開されている範囲では具体的なベンチマーク数値が途中で切れており、**どのベースラインに対してどれだけ勝ったかは本文で確認が必要**。「rare balance」という表現は定量的裏付けを見るまで額面通りに取らないこと。

## 追っているプロジェクトへの適用可能性

**P3 (次世代アーキテクチャ) — 最も効く。**
VLA (Vision-Language-Action; 視覚と言語を入力に受けて行動を出力するモデル) の視覚側をどう組むかは P3 の中心的な未解決点で、この論文はそこに2つの具体案を出している。(1) I3D-ViT による**動画エンコーダの安価な立ち上げ** — 画像で事前学習した ViT 資産を捨てずに時間方向へ拡張できる。(2) Adaptive Frame Resolution — **車載カメラのストリームは典型的な streaming video perception そのもの**で、「直前フレームは高解像度、遠い過去は低解像度」といった計算配分は運転 E2E の文脈長問題に直接刺さる。

ここは 07-19 の LongStraw / 07-18 の RoboTTT と同じ問題系に属する。あの2本は**長 context のコストをどの段階 (学習時 replay / 推論時 fast weights) で払うかの議論**だったが、VideoChat3 は**そもそも入力トークンを入口で減らす**という第三の答えを出している。3本を「長い時系列入力のコストをどこで畳むか」の設計空間として並べると P3 のアーキ選定メモが一段強くなる。

**P2 (FM 蒸留 + 適合) — 実務的に効く。**
fully open であることが蒸留の観点で価値を持つ。**teacher の重みだけでなく学習データと学習戦略まで手に入る teacher は蒸留の設計自由度が段違い** — teacher が何を見て学んだかが分かるので、student のデータ選択や capacity gap (teacher と student の表現力差が大きすぎて蒸留が失敗する現象) への対策を根拠を持って設計できる。加えて3つのデータセットが用途別に分かれているため、**ドメイン適合の際にどのサブセットを残すかの実験がしやすい**。

**P1 (Planner AI + 評価) — 間接的。**
直接のプランナー技術ではないが、走行ログから「何が起きているシーンか」を自動でラベリングする用途 (シナリオマイニング) には使える可能性がある。ただし本命の適用先ではないので、P3/P2 の検証が済んでから触る優先度。

## 試すべき実験アイデア

1. **Adaptive Frame Resolution の運転データへの移植可否を測る。** 自社の走行動画で「フレームごとに解像度を変えたときの下流タスク性能の劣化カーブ」を取る。**どこまで解像度を落とせるかは、遠方の小さい物体をどれだけ見るかに完全に依存する**ので、汎用動画での結果はそのまま転用できない。ここは 1〜2 日で足切りできる。
2. **I3D-ViT の inflation を手元の画像エンコーダで再現する。** 既に持っている 2D の視覚バックボーンを時間方向へ inflate して、動画をゼロから学習した場合とのコスト/性能比を取る。**うまくいけば動画エンコーダの学習予算が丸ごと浮く**ので、期待値が最も高い実験。
3. **teacher としての適性評価 (P2)。** 公開データの3分割のうち、自社ドメインに近いのがどれかを埋め込み距離で測り、蒸留時の混合比を決める材料にする。fully open だからこそ実行できる分析。
4. **「入口で減らす vs 途中で畳む」の比較表。** VideoChat3 (入力トークン削減) / RoboTTT (推論時圧縮) / LongStraw (学習時 replay) を同じ軸で並べ、P3 のアーキ選定メモに追記する。07-19 に作りかけた比較表の3つ目の列として入れるのが自然。
