# The Mirage of Optimizing Training Policies: Monotonic Inference Policies as the Real Objective for LLM RL

- **arxiv-id**: 2606.29526 ／ **topic**: sns_wildcard ／ **project**: EXPLORE (P2 に接続) ／ **relevance**: 3/5 ／ **hf_upvotes**: 111(本日候補中の最高)
- **URL**: https://arxiv.org/abs/2606.29526

## 何を解決する論文か
LLM を RL (Reinforcement Learning; 強化学習) で post-training(事前学習後の追加学習。ここでは推論能力を伸ばす仕上げ工程)する際、学習が不安定で崩壊(collapse; 性能が突然壊れる現象)しやすい。主因の一つが **training-inference mismatch**: LLM は生成の高速化用に inference engine を、勾配計算の精度用に training engine を **別々に**使うため、パラメータを同期しても同じ trajectory(生成系列)に対して両エンジンが異なる確率を返してしまう。これが常在的な off-policyness(学習に使うデータの分布が、最適化対象の方策の分布とずれている状態)を生んで学習を汚染する。本論文は、既存研究が見落としてきた **目的のズレ** を指摘する — training engine 側の方策をうまく更新できても、実際に deploy される inference engine 側の方策が良くなるとは限らない。そこで「inference 側の方策の単調改善」を直接の最適化目標に据える枠組みを提案する。

## 手法の核心
- **MIPI (Monotonic Inference Policy Improvement)**: 最適化すべきは training policy ではなく **inference policy**(=実際に本番で使う方)であり、その inference policy が更新ごとに単調に良くなることを原理として掲げる。従来の「training policy を最適化すれば十分」という前提を否定するのが着眼点。
- **MIPU (Monotonic Inference Policy Update)**: MIPI を実装する 2 段階の RL フレームワーク。
  1. **sampler-referenced candidate updates**: サンプリング(生成)に使ったエンジンを基準として候補更新を構成する。
  2. **selective acceptance**: 更新候補を無条件に適用せず、**inference-side gap proxy**(inference 側の確率ズレを近似する指標)を見て、inference policy を実際に改善しそうな同期済み候補だけを **選択的に受理**する。悪化させる更新を弾くことで単調性を担保する。
- **gap proxy による受理判定**が肝: train/infer の確率ギャップを直接測るのは高コストなので、その代理指標で「受け入れてよい更新か」を安価に判定する設計。
- **主要結果**: 2 つのモデルスケールで、mismatch が大きい条件下でも平均 reasoning 性能と学習安定性の両方が向上。「安定化のために off-policyness を補正する」従来路線に対し、「そもそも最適化対象を inference policy に取り替える」という視点の転換を示した。

## P2(Foundation Model 蒸留 + 適合)への適用可能性
- P2 の relevance_criteria は「再現可能な recipe / 新しい loss」を高優先に挙げており、MIPU は **新しい policy 最適化目標(=新 loss/受理則)** に該当する。蒸留後・適合後のモデルを RL で仕上げる工程を回すなら、学習の崩壊を防ぐ実務ノブとして直接効く。
- **本質的な学び**は RL 固有ではない: 「**最適化しているオブジェクト(training engine の重み)と、実際に価値が出るオブジェクト(deploy 時の挙動)がズレていないか**」という問いは、蒸留・量子化・PEFT すべてに一般化する。例えば蒸留時に teacher-student を fp32 で loss を取っても、student を int8 で deploy するなら「deploy 時挙動の単調改善」を見るべき、という MIPU と同型の落とし穴が P2 に存在しうる。
  - PEFT = Parameter-Efficient Fine-Tuning(全パラメータでなく少数の追加/選択パラメータだけ更新する適合手法。LoRA 等)
- **selective acceptance という設計パターン**は汎用的: 適合の各更新ステップで「本当に deploy 指標を改善するか」を安価な proxy で判定し、悪化更新を弾く。少データ適合(P2 が挙げるドメインギャップ克服)で過学習・崩壊を抑える汎用テンプレになりうる。
- 注意: 対象は LLM の reasoning RL であり、自社の蒸留/適合パイプラインへの適合は **間接的**。ただし「train と deploy のエンジン差が off-policyness を生む」という指摘は、量子化推論や別バックエンド deploy を前提とする現場で **見落とされがちな実害** であり、探索枠としての学習価値は高い。

### 試すべき実験アイデア
1. **train/deploy エンジン差の実測**: 自社の適合/蒸留パイプラインで、学習時エンジン(高精度)と deploy 時エンジン(量子化/別バックエンド)に **同一入力**を通し、出力分布(logits/確率)のギャップを測る。無視できない差があれば MIPU 型の是正余地がある、という健康診断。
2. **selective acceptance の移植**: 適合の更新を受け入れる前に、deploy 相当の推論(量子化込み)で検証指標を測る安価な proxy を作り、**deploy 指標を悪化させる更新ステップを棄却**する gating を fine-tuning ループに挟む。少データ適合の安定性が上がるかを ablation。
3. **蒸留の目的関数の張り替え**: teacher-student 蒸留で loss を training 精度で取るのではなく、**student の deploy 時挙動**(量子化後)で単調改善を追う目的に張り替え、capacity gap(teacher と student の容量差)が大きい設定で性能が改善するか検証。
