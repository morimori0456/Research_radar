## 1. システム全体像 — 5分で理解する

### 何のためのツールか

「最新研究のインプット」「世の中の課題理解」「スキル成長」「人生計画との整合」を、
**あなたが意識しなくても毎日・毎週・毎月・毎四半期、自動で回す**ためのシステム。

設計の核は2つ:

1. **カスケード** — 安い処理 (スクリプト) で広く集め、賢い処理 (Claude) で絞り、
   あなたは最後の判断だけをする。読む量は1日3件に固定される
2. **LIFE_PLAN への従属** — 全ループが LIFE_PLAN.md (10年計画) を読んでから動くので、
   日々の推薦が自動的に現フェーズの目標に寄る

### ループの階層

```
LIFE_PLAN.md (北極星: 10年計画)
    ▲
四半期レビュー ── OKR設定 ──▶ 月次レビュー ──▶ 週次レビュー ──▶ あなたの毎日
(計画との整合)    (方向修正)      (定着・実行指令)
    ▲                ▲               ▲
課題レーダー(水)   技術radar(毎晩)   │
「解くべき問題」   「解き方」        │
                                    ▼
                            精読1本・実験1件・出力 (成長が起きる瞬間)
```

### 自動実行スケジュール (cron)

| 時刻 | ループ | 役割 |
|---|---|---|
| 毎晩 3:00 | 日次 radar | arXiv+SNS注目度収集 → 選別 → ブリーフ生成 → push |
| 水曜 21:00 | 課題レーダー | 業界動向 → 課題カード3枚 |
| 日曜 20:00 | 週次レビュー | テーマ統合 / 精読推薦 / 実験棚卸し / 記憶チェック |
| 毎月1日 20:00 | 月次レビュー | スキルマップ / 基礎テーマ / ポートフォリオ監査 / 論文PL |
| 1,4,7,10月1日 21:30 | 四半期レビュー | Phase進捗 / 決断材料 / OKR / LIFE_PLAN更新 |

## 2. ファイルマップ — 何がどこにあるか

```
research_loop/
│
│  ── 設定 (あなたが編集する) ──────────────────────
├── topics.yaml            # 追う分野・キーワード・選別基準・上限
├── LIFE_PLAN.md           # 10年計画。「本人メモ」欄はあなた専用
│
│  ── 状態 (ループが更新、あなたは読む+時々直す) ──
├── RESEARCH_PIPELINE.md   # 論文アイデア在庫と締切カレンダー
├── SKILL_MAP.md           # スキル自己評価 (月次が根拠付きで更新)
├── surveys/               # Living Survey: テーマ別に育つ個人サーベイ (週次が統合)
├── decisions/             # Decision Journal: 意思決定の記録 (あなたが書く、四半期が答え合わせ)
│
│  ── スクリプト (通常触らない) ────────────────────
├── fetch_candidates.py    # Stage1: arXiv + HF Daily Papers 収集
├── deep_research.sh       # Stage2: Claude選別 + ブリーフ生成
├── run_research.sh        # 日次オーケストレータ
├── problem_radar.sh       # 課題レーダー (水曜)
├── weekly_review.sh       # 週次レビュー (日曜)
├── monthly_review.sh      # 月次レビュー (1日)
├── quarterly_review.sh    # 四半期レビュー (1,4,7,10月)
├── pull_experiments.sh    # 下流: ブリーフ→実験雛形 (別マシンで使う)
│
│  ── 出力 (ループが生成) ──────────────────────────
├── briefs/YYYY-MM-DD/     # 日次: 論文ブリーフ + DIGEST.md + rejected.md
├── problems/YYYY-Wxx.md   # 週次: 課題カード
├── weekly/YYYY-Wxx.md     # 週次レビュー
├── monthly/YYYY-MM.md     # 月次レビュー
├── quarterly/YYYY-Qx.md   # 四半期レビュー
└── candidates.json        # 中間ファイル (gitには入らない)
```

**編集の原則**: `topics.yaml` と `LIFE_PLAN.md` の本人メモ欄はあなたの領域。
出力ファイルはループの領域 (手で直しても次回上書きされうる)。
`RESEARCH_PIPELINE.md` / `SKILL_MAP.md` は共有領域 — ループが更新するが、
あなたが直接直してもよい (次回のループはあなたの編集を前提に動く)。

## 3. あなたの操作 — 日/週/月/四半期にやること

### 毎朝 (5分)

1. `briefs/今日の日付/DIGEST.md` を開く
   - ローカル: `cat briefs/$(date +%F)/DIGEST.md`
   - スマホ: GitHub の Research_radar リポジトリから
2. 冒頭の **TOP3 だけ読む**。気になる1件だけ個別ブリーフへ
3. 「これは実験になりそう」と思ったらメモ (深追いは週末)

DIGEST がなければ夜間実行が失敗している → [トラブルシューティング](#7-トラブルシューティング)へ。

### 木曜朝 (3分)

`problems/今週.md` (水曜夜生成) の課題カード3枚に目を通す。
読むだけでよい。蓄積が Phase 3 の「課題ドメイン」に育つ。

### 日曜夜 (15分)

1. 20時に生成される `weekly/今週.md` を読む
2. **記憶チェック5問に答える** (思い出せなければ該当ブリーフを再読)
3. 「今週の精読1本」→ 土曜の予定に入れる
4. 「実験アイデアの棚卸し」→ 未着手から1件選び、水曜夜の予定に入れる
5. 「出力キュー」→ ML_report に書くか判断
6. **末尾の「身体基盤の自己申告欄」に睡眠・運動を1行ずつ記入** (詳細は HABITS_GUIDE.md)
7. 気が向いたら `surveys/` の育ち具合を眺める (読むだけでよい)

### 重要な決定をしたとき (随時・5分)

`decisions/_template.md` をコピーして `decisions/YYYY-MM-DD_名前.md` を書く。
予測と確信度が本体 (四半期レビューが答え合わせする)。対象は月2-3件の重要な決定のみ。

### 週の中でやること (合計 週4時間)

- **土曜 90分**: 精読1本 (原論文を「読むときの問い3つ」に答えながら)
- **水曜夜 90分**: 実験1件に着手 (完了でなくてよい)

### 毎月1日以降 (30分)

`monthly/今月.md` を読む:
- 「実績 (事実ベース)」— 未実行の数から逃げない
- SKILL_MAP の変化を確認
- 「来月の基礎テーマ」→ 学習プランをカレンダーに落とす
- 「今月公開すべき1本」→ やるか捨てるか決める
- 「ループ自体の監査」→ 削減提案があれば従うか判断

### 四半期 (1時間 — 1,4,7,10月の初週)

`quarterly/今期.md` を読み、以下を**自分の手で**行う:
1. OKR (最大3つ) を承認 or 修正 — 以降の全ループがこれに従う
2. 「31歳の決断ポイント」の判断材料を確認
3. LIFE_PLAN.md への提言があれば、本文を自分で書き換える
   (現在地セクション以外はループは書き換えない設計)
4. 「本人メモ」欄を更新 (家族・資金など数字に出ない制約)

## 4. 各ループの詳細リファレンス

### 4.1 日次 radar (`run_research.sh`)

```
fetch_candidates.py ─▶ candidates.json ─▶ deep_research.sh ─▶ briefs/ ─▶ git push
```

**Stage 1 (`fetch_candidates.py`)** — AI不使用:
- `topics.yaml` の各トピックで arXiv API を検索 (`lookback_days` 日以内の新着)
- Hugging Face Daily Papers API からコミュニティ注目度 (upvotes) を取得し
  各候補に `hf_upvotes` を注釈
- 分野外でも upvotes≥10 の注目論文を wildcard 候補 (最大3件) として追加

**Stage 2 (`deep_research.sh`)** — Claude:
- 各候補を relevance_criteria で 0-5 点評価
- quota (トピック別) と max_deep_per_day (全体) の範囲で選抜
- 選抜分: 1ページブリーフ (`briefs/日付/arxiv-id.md`)
- 落選分: 理由付きで `rejected.md`
- 全体: `DIGEST.md` (TOP3 + 一覧 + プロジェクト別要点)
- wildcard は「学び価値があれば最大1件/日」
- スタイル: 専門用語は英語のまま、初出に説明、略語は展開

**出力の読み方**:
- `score 5` = 今週試す価値あり / `4` = 精読候補 / `3` = 記録しておく程度
- `hf` 列 = SNS注目度。高い≠重要 (副次シグナル)
- rejected.md は選別の妥当性を疑うときに見る

### 4.2 課題レーダー (`problem_radar.sh`)

技術 radar が「解き方」を集めるのに対し、これは「解くべき問題」を集める。
業界ニュース・事故・規制・資金調達を調べ、課題カード3枚/週を生成:

- 誰が困っているか / 規模感
- なぜ未解決か (技術? コスト? 規制? 商流?)
- あなたの技術との接続
- 検証の最初の一歩 (1週末粒度)

「ストックへの示唆」セクションが繰り返し現れるテーマを検出する。
**同じテーマが数回続いたら、それがあなたの Phase 3 の課題ドメイン候補**。

### 4.3 週次レビュー (`weekly_review.sh`)

今週の全ブリーフを読み直して生成。6セクション:

| セクション | 使い方 |
|---|---|
| 1. テーマ統合 | 週単位の潮流。流し読みでよい |
| 2. 精読1本 | **土曜に必ず実行**。問い3つに答える形で読む |
| 3. 実験棚卸し | 未着手リスト。1件選んで水曜夜へ |
| 4. 記憶チェック5問 | **日曜夜に答える**。定着の要 |
| 5. topics.yaml 提案 | 採用するかはあなたの判断 |
| 6. 出力キュー | ML_report に昇格させるテーマ。アウトライン付き |

LIFE_PLAN と最新の quarterly OKR を読んでから推薦する設計なので、
推薦が計画とずれていると感じたら LIFE_PLAN 側を疑う (計画が古い可能性)。

### 4.4 月次レビュー (`monthly_review.sh`)

7セクション: 実績カウント / SKILL_MAP更新 / 基礎テーマ /
ポートフォリオ監査 / 外部フィードバック機会 / 論文パイプライン更新 / ループ監査。

特に重要な設計:
- **SKILL_MAP は根拠なしの評価上昇を禁止** — 「何をやったか」が書けない項目は上がらない
- **基礎テーマは月1つだけ** — radar の最新論文バイアスを補正する唯一の仕組み
- **ループ監査** — 形骸化したセクションの削減を提案する。システムの肥大化防止弁

### 4.5 四半期レビュー (`quarterly_review.sh`)

唯一 LIFE_PLAN.md を書き換える権限を持つループ (現在地セクションのみ)。
6セクション: Phase進捗判定 / 決断材料 / 論文絞り込み /
課題ドメイン収束チェック / 来期OKR / LIFE_PLAN への提言。

**OKR は全ループの上位指令になる**ので、承認は真剣に。
3つを超える提案が来たら削らせる (全部追うと全部落ちる)。

## 5. カスタマイズ方法

### 5.1 追う分野を変える (`topics.yaml`)

```yaml
- name: トピック名          # 一意の識別子
  project: P1              # ブリーフに表示されるラベル
  arxiv_categories: [cs.RO, cs.AI]   # arXivカテゴリ (広めに)
  keywords:
    - "motion planning"    # abstract に含まれる語句 (英語)
  relevance_criteria: >
    選別基準を日本語で。Claude はこれを読んで0-5点を付ける。
    「〜は高優先」と書くと効きやすい。
  quota: 2                 # このトピックから1日最大何件深掘りするか
```

変更のコツ:
- **候補が少なすぎる** → keywords を広げる or カテゴリを足す (P3 で実証済み:
  keywords 追加で 0件→15件)
- **ノイズが多い** → keywords はそのまま、relevance_criteria に除外条件を書く
  (収集は広く、選別で絞るのが正しい形)
- **編集後は必ず**: `python3 fetch_candidates.py` で候補数を確認してから commit

```bash
# 変更を反映
cd ~/w/research_loop
$EDITOR topics.yaml
python3 fetch_candidates.py          # 候補数の確認
jq '.candidates | group_by(.topic)[] | "\(.[0].topic): \(length)"' candidates.json
git add topics.yaml && git commit -m "topics: <変更内容>" && git push
```

### 5.2 実行時刻を変える (crontab)

```bash
crontab -e
# 例: 日次を4時に → 「0 3 * * *」を「0 4 * * *」に
# 書式: 分 時 日 月 曜日 (曜日: 0=日曜)
```

### 5.3 出力のスタイルや構成を変える (プロンプト編集)

各 `.sh` の `PROMPT=$(cat <<EOF ... EOF)` ブロックが Claude への指示。
セクションの追加・削除・文体変更はここを直接編集する。

過去の実例: 「専門用語は英語のまま+初出に説明」はこの方法で全プロンプトに追加した。
編集後は `bash -n スクリプト名` で構文確認 → commit & push。

### 5.4 深掘り件数を変える

`topics.yaml` の `global.max_deep_per_day` (全体上限) と各トピックの `quota`。
読む時間が足りないなら減らす。上限に毎日張り付くなら選別基準が甘い。

## 6. 手動実行コマンド集

```bash
cd ~/w/research_loop

# 日次を今すぐ回す (全ステージ)
./run_research.sh

# 収集だけ試す (Claude を使わない、無料)
python3 fetch_candidates.py
jq '.candidates | length' candidates.json

# 深掘りだけやり直す (candidates.json が必要。同日分は上書き)
./deep_research.sh

# 週次/月次/四半期/課題レーダーを今すぐ
./weekly_review.sh
./monthly_review.sh
./quarterly_review.sh
./problem_radar.sh

# ログを見る
tail -50 ~/research_loop.log

# 今日の DIGEST を読む
cat briefs/$(date +%F)/DIGEST.md

# 過去のブリーフを検索 (例: capacity gap に触れたブリーフ)
grep -rl "capacity gap" briefs/
```

## 7. トラブルシューティング

### 朝、DIGEST がない

```bash
tail -30 ~/research_loop.log     # 昨晩3時のログを確認
```

| ログの症状 | 原因と対処 |
|---|---|
| ログ自体がない | PCが夜間スリープしていた → 常時起動にするか cron 時刻を変更 |
| `[fetch] ... error` | arXiv/HF API の一時障害 → `./run_research.sh` で手動再実行 |
| `claude: command not found` | crontab の PATH 行が消えている → `crontab -e` で先頭に `PATH=/home/jetson/.local/bin:/usr/bin:/bin` を戻す |
| `[push] push 失敗` | 認証切れ → 下記「push が失敗する」 |
| Claude の rate limit / quota | その日は手動で `./deep_research.sh` を時間を置いて再実行 |

### push が失敗する

```bash
cd ~/w/research_loop && git push origin main   # 手動で試す
```
- 認証を聞かれる → トークン期限切れ。GitHub で新トークン発行
  (Fine-grained: Research_radar を選択 + Contents: Read and write)。
  push 時のパスワード欄に入力すれば `credential.helper store` が再記憶する
- `rejected (fetch first)` → 別マシンで push した →
  `git pull --rebase origin main` してから push

### ブリーフの品質が悪い

- **的外れな論文が選ばれる** → `rejected.md` と見比べ、relevance_criteria に
  「〜は対象外」と明記する
- **説明が足りない/くどい** → `deep_research.sh` のスタイル規則を調整
- **同じ論文が連日出る** → 仕様 (lookback_days=2 のため)。気になるなら 1 に減らす。
  ただし週末の取りこぼしとトレードオフ

### 週次/月次が生成されない

前提データ不足の可能性 (例: 週次は直近7日のブリーフが必要)。
ログに `直近7日のブリーフがない` 等が出ていれば、日次が数日止まっていた証拠。
日次を復旧してから手動実行すればよい。

### Claude の応答が途中で切れている / ファイルが空

`claude -p` のタイムアウトや一時エラー。該当スクリプトを手動再実行。
同日分は上書き生成されるので安全。

## 8. メンテナンス

| 頻度 | 作業 | 所要 |
|---|---|---|
| 週1 (日曜) | 週次レビューの「topics.yaml への提案」を見て採否判断 | 2分 |
| 月1 | 月次の「ループ監査」を読み、形骸化セクションの削減提案に従うか判断 | 5分 |
| 月1 | `RESEARCH_PIPELINE.md` の締切カレンダーが古くないか一瞥 | 2分 |
| 四半期 | LIFE_PLAN.md の本人メモ欄を更新 | 10分 |
| トークン期限時 | GitHub トークン再発行 (期限は発行時に自分で設定) | 5分 |
| 随時 | `briefs/` が肥大化したら古い月を `archive/` へ移動 (任意) | — |

システム全体は git 管理なので、**このリポジトリを clone すれば別マシンに完全移行できる**
(crontab と credential だけ再設定)。

## 9. 禁止事項と注意

1. **業務の機密情報をこのリポジトリに入れない** — ここは公開情報 (arXiv/HF/ニュース) だけを
   扱う場所。境界が崩れるとシステム全体が使えなくなる
2. **topics.yaml の編集はこのマシン (NUC) に統一** — 複数マシンで編集すると git が
   枝分かれする。別マシンで直したくなったら、必ず先に `git pull`
3. **トークンをチャットやファイルに貼らない** — 認証は端末のプロンプトで入力する
4. **出力ファイルの手動編集は上書きされる前提で** — 恒久的な指示はプロンプト
   (スクリプト内) か LIFE_PLAN.md に書く
5. **ループを増やしすぎない** — 新しいループは月次の「ループ監査」が必要と言ってから。
   維持コストは複利で効いてくる
6. **最重要**: このシステムは成長の機会を目の前に置くだけ。
   **精読1本 (土曜) と実験1件 (水曜) をやるかどうかだけが、成長の全て**
