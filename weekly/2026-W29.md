## 4. 記憶チェック (5問)

1. **Q**: 蒸留データ生成で失敗サンプルを「gold answer を見せて chain を書かせる」で救済すると何が起きる? 症状は?
   **A**: 答えから逆向きに正当化 (rationalize backward) した chain になり、**correctness filter では検出できない形で**データが劣化。最難問で最大 **~27pt** の精度低下。計測可能な症状は **early final-answer statement (推論の早い段階で最終答えを言ってしまう)** の出現率。処方は **answer-blind で生成せよ**。
2. **Q**: RoboTTT が context 8K timestep でも推論レイテンシを増やさない仕組みと、そこから見えた新しい scaling 軸は?
   **A**: 履歴を attention でなく **fast weights (推論時も勾配で更新される重み) の recurrent state に圧縮**するので更新が O(1) 的。新しい軸は「**事前学習の context 長**を伸ばすと閉ループ性能が単調改善する」。
3. **Q**: TerraZero が実走行ログに与えた役割は? デモは何件使った?
   **A**: 実ログは **地図形状の供給源としてのみ**使い、道路利用者・信号は手続き的に配置。**人間デモは0件**で self-play RL のみ。それで long-tail ベンチ InterPlan の学習型プランナー首位、左側通行まで創発。
4. **Q**: FlowWAM が action 表現に optical flow を選ぶ理由は(2つ)?
   **A**: (1) flow 動画は **RGB 動画と同じ形式**なので同一の事前学習動画生成器で dual-stream 生成でき、policy mode と world-model mode を1モデルで回せる。(2) flow は**生動画から自動抽出できる**ので action ラベルなしの大規模動画で事前学習できる。
5. **Q**: UMoE が SFT 前に行う処理と、据え置かれるものは?
   **A**: ドメインデータで測った **saliency の低い expert を prune → 摂動ベースの複製で元の数まで regrow → 通常 SFT**。**パラメータ数と推論コストは完全に不変**。ドメインごとのハイパラ調整なしの単一 recipe で 2アーキ×5ドメイン×12ベンチ全勝。

**テーマ: 「推論ステップを削る生成 — diffusion から one-step drift / schedule 変更へ」(R1 × R3)**
選定理由: 今週最も強い横断シグナル。VSFM (再学習なしで積分グリッドを変えるだけ) / DRIFT (軌道 latent で
one-step drift、48 proposal を1パス・10.82ms) / DriftWorld (一発生成で diffusion 比 17倍、action search が実用域) /
FlowWAM (同一生成器の2モード運用) が、P1 のプランナーと P3 の world model で**同じ答え**に到達した。
車載ではNFE (Number of Function Evaluations; サンプリング時のネットワーク評価回数) がそのままレイテンシ予算になるため、
**「品質 vs 推論ステップ数」のトレードオフは数ヶ月どころか数年残る実務の軸**。小規模で完全再現でき notebook 向き。

レポート・アウトライン(md + 実行可能 notebook):
- **§1 なぜ NFE が設計変数か**: 生成型プランナー/world model のレイテンシ予算と NFE の関係を定量化
- **§2 一発生成の系譜**: diffusion (多ステップ denoising) → flow matching → drifting の定式化を1枚に整理
- **§3 学習しないで削る**: 推論時 τ-schedule 変更 (VSFM)。既存学習済みモデルに追加コストゼロで効く範囲
- **§4 学習して削る**: one-step drift (DRIFT / DriftWorld) と、失う多様性・多峰性の代償
- **§5 多仮説をどう畳むか**: proposal 集合 → 単一出力の集約を**品質ラベルなしで**学習する (DRIFT の Aggregation Head)
- **§6 実務指針**: レイテンシ予算から逆算する「どこまで削ってよいか」の判断表

notebook で実証すべき3点:
1. **NFE を横軸にした品質カーブが手法ごとに違う形をする** — 小規模 flow matching で NFE 4/8/16/32 の
   品質を測り、多ステップ前提の手法が低 NFE で崩れる点を可視化する
2. **推論時 τ-schedule (非一様な時刻グリッド) の変更だけで、再学習なしに低 NFE 域の品質が改善する** —
   uniform vs braking プロファイルで VSFM の主張を再現し、Euler の local truncation error の説明と突き合わせる
3. **品質ラベルなしの集約ヘッドが、best-of-N や単純平均に勝つ** — 多峰な toy 分布で候補集合を生成し、
   平均 (モード間の谷に落ちる) / ランダム選択 / 学習した集約を比較して DRIFT の設計を最小再現する
